# Primary references

Use these sources as decision aids, not as substitutes for repository evidence. Accessed 2026-07-13.

| Source | Scope and currentness | Use in this skill |
| --- | --- | --- |
| [ISO/IEC/IEEE 42010:2022](https://www.iso.org/standard/74393.html) | ISO architecture-description standard, edition 2, published 2022-11 | Stakeholders, concerns, viewpoints, views, and architecture descriptions |
| [SEI Architecture Tradeoff Analysis Method](https://www.sei.cmu.edu/library/architecture-tradeoff-analysis-method-atam/) | Carnegie Mellon SEI method page, updated 2026-05-11 | Structured evaluation of business drivers, quality scenarios, risks, sensitivities, and tradeoffs |
| [SEI Quality Attributes](https://www.sei.cmu.edu/library/quality-attributes/) | SEI reference collection | Quality-attribute reasoning and scenario-based analysis |
| [SEI Quality Attribute Workshop](https://www.sei.cmu.edu/library/quality-attribute-workshop-collection/) | SEI method collection | Stakeholder-driven quality scenario elicitation |
| [C4 model](https://c4model.com/) and [diagram guidance](https://c4model.com/diagrams) | Maintained official model site | Context and container views; avoid unnecessary code-level diagrams |
| [arc42 overview](https://arc42.org/overview) | Maintained official architecture documentation template | Tailorable architecture documentation and cross-cutting concepts |
| [Architecture Decision Records](https://adr.github.io/) | Community hub linking the original and maintained ADR formats | Durable decision context, consequences, status, and supersession |
| [Architectural fitness function](https://www.thoughtworks.com/en-ca/radar/techniques/architectural-fitness-function) | Thoughtworks Technology Radar technique | Objective architecture checks through tests, metrics, and monitors |
| [Hexagonal Architecture](https://alistair.cockburn.us/hexagonal-architecture) | Alistair Cockburn's original 2005 description | Ports and adapters around application policy |
| [Monolith First](https://martinfowler.com/bliki/MonolithFirst.html) | Martin Fowler, 2015; experience-based and explicitly contextual | Evidence against premature service decomposition, not a universal rule |
| [Microservice tradeoffs](https://martinfowler.com/articles/microservice-trade-offs.html) | Martin Fowler, 2015; practitioner synthesis | Operational and distributed-system costs of service boundaries |
| [What do you mean by event-driven?](https://martinfowler.com/articles/201701-event-driven.html) | Martin Fowler, 2017 | Distinguish event notification, state transfer, event sourcing, and CQRS |
| [AWS Well-Architected pillars](https://docs.aws.amazon.com/wellarchitected/latest/framework/the-pillars-of-the-framework.html) | Current AWS framework; provider-specific | Operational excellence, security, reliability, performance, cost, and sustainability prompts |
| [Google Cloud Well-Architected Framework](https://docs.cloud.google.com/architecture/framework) | Current Google Cloud framework; reviewed 2026-01-28 | Cross-cutting reliability, security, cost, performance, operations, and sustainability prompts |
| [Azure Well-Architected pillars](https://learn.microsoft.com/en-us/azure/well-architected/pillars) | Current Microsoft Azure framework; provider-specific | Reliability, security, cost, operational excellence, and performance prompts |
| [Google SRE book](https://sre.google/sre-book/table-of-contents/) | Google-authored operational reliability reference | SLOs, monitoring, overload, release, and operational design concepts |

Provider frameworks reflect their platforms and incentives. Use them as checklists only after defining the system's actual constraints.
